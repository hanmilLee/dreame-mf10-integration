## Decisioni

1. **`_prop()` helper in fan.py**: legge `MF10_PROPERTY_MAP` per costruire
   il payload `{siid, piid, value}` — niente siid/piid hardcodati nel corpo
   dei metodi. Cambiamenti alla property map si propagano automaticamente.
2. **Batch `set_properties` in `turn_on`**: power + speed e/o mode inviati
   in una sola chiamata API quando entrambi forniti. Riduce latenza percepita.
3. **`set_percentage(0)` → `turn_off()`**: `percentage_to_ranged_value`
   con percentage=0 darebbe speed=0 (fuori range); delegare a `turn_off`
   è più corretto e coerente con il comportamento standard HA.
4. **OFF_BEHAVIOR_SOFT deferred**: `turn_off` usa sempre `power=2` (off reale).
   Soft-off (sleep speed 1) rimandato a M4 per evitare deep standby
   non recuperabile — TODO già citato nel CLAUDE.md.

## Risultati test funzionale (sessione continuazione)

### Comportamento cloud relay confermato empiricamente

- `get_properties` (polling): funziona sempre — legge dalla cache server Dreame.
- `set_properties` mode/speed/oscillation: **arriva sempre al device via cloud**,
  anche quando il device risulta spento (`power=2`). Il beep fisico conferma la
  consegna del comando; il problema ON/OFF non è quindi la raggiungibilità del
  device via cloud.
- `set_properties` power=2 (real off): **funziona** ma causa disconnessione WiFi
  → device irraggiungibile via cloud fino alla riaccensione fisica.
- `set_properties` power=1 (turn on da off): ricevuto (beep) ma **ignorato** dal
  device — stesso pattern AP10 (`MUST use toggle action`). Action siid=2 aiid=3
  fallisce con 80001 (richiede risposta real-time, non supportata).
- `action` (qualsiasi): 80001 — le action richiedono risposta sincrona dal device,
  il relay REST non riesce a garantirla.

### Correzione diagnosi canale cloud

La precedente ipotesi di una "finestra MQTT" di circa 20 secondi era errata.
I comandi `mode`, `fan_speed` e `oscillation` inviati da HA vengono ricevuti dal
device anche quando è spento: il beep fisico arriva sempre. Il problema residuo
è specifico della semantica power: il device riceve comandi property, ma non
interpreta `mode`/`speed` come richiesta di accensione reale.

Rilettura snapshot `research/snapshots/`:
- `after-power-on` → `after-power-off`: `(2,1): 1→2`, `(2,4): 8→6`.
- `after-power-off` → `after-mode-sleep`: `(2,1): 2→1`, `(2,3): 0→2`,
  `(2,4): 6→1`.

Quindi l'app, almeno in quel test, accende il device anche tramite cambio mode
da stato off. Test mancante: replicare esattamente `set_properties(mode=2)` da
HA/CLI partendo da `power=2`, poi leggere subito `power`.

### Errore F2

Dopo re-pairing WiFi il device mostra F2 (cloud relay instabile). Ha impedito
test funzionale completo del flusso HA. Il test con soft-off è pending.

### Soft-off implementato (commit 0b5786a)

`turn_off` → mode=Sleep + speed=1 (power rimane 1) → device resta connesso WiFi.
`is_on` → False se power=1 AND mode=Sleep AND speed≤1.
`turn_on` → power=1 + mode=AI (esce dal soft-off).
Questo evita il real-off che causa WiFi disconnect.

## Scoperta display codes (sessione continuazione)

I codici `F1`–`F4` sul display fisico **non sono errori** — sono indicatori di modalità:

| Display | Modalità | MiOT value |
|---------|----------|------------|
| `F1` | AI (automatica) | 0 |
| `F2` | Sonno (Sleep) | 2 |
| `F3` | Naturale | 7 |
| `F4` | Potente | 1 |
| numero (es. `5`) | Manuale — velocità corrente | 3 |

Tutti i riferimenti precedenti a "errore F1/F2" nel log erano in realtà stati
operativi normali del device (F1 = AI mode all'avvio, F2 = soft-off attivo).
Documentato in `docs/property_map.md`.
